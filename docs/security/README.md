# Security

`docs/security/` is the supply-chain ledger for agentic-playbook. It holds the **AI Bill of Materials** (`ai-bom.json`) and any other artifacts the supply-chain gate emits. Per ADR-0047.

## What ships here

| File | What it is | Generated by |
|---|---|---|
| `ai-bom.json` | AI Bill of Materials. One entry per imported skill, MCP bundle, or other external component the playbook ships. Records source, version (where pinned), `vetted_as_of` date, and component kind. | `scripts/security/ai_bom.py` (called by `make check` + `make audit`). |

## What an AI BOM is

An AI Bill of Materials is the AI-platform equivalent of a Software Bill of Materials (SBOM). For each external skill, subagent, MCP bundle, or other shipped third-party content, the BOM records:

- **`name`** and **`kind`** (e.g. `imported_skill`, `mcp_bundle`)
- **`source`** (the upstream URL or org slug the content came from)
- **`path`** (where it lives inside this repo)
- **`version`** (if the upstream has a pinned version)
- **`vetted_as_of`** (the date the maintainer last reviewed the import)

The point: a downstream consumer can audit exactly what they're installing without walking the tree by hand. A future CVE feed can diff against this BOM to surface affected installations.

## How the BOM is consumed

- **`make check`** regenerates the BOM as part of its quality pipeline. Idempotent: a clean repo with no upstream changes produces a bit-identical BOM file (the `generated_at` timestamp is preserved when components are unchanged, per the BOM idempotency contract in `scripts/security/ai_bom.py`).
- **`make audit`** is the block-by-default external-skill security gate. It walks newly-imported skills and refuses to land them in the playbook unless they pass the audit. The BOM is updated to reflect the final state.
- **`docs/atlas/`** reads the BOM at build time to render the per-skill `vetted: <date>` badge.

## Where the threat model lives

ADR-0047 spells out the threat model and the gate's design:

- **Document-Driven Implicit Payload Execution (DDIPE)** detector under `scripts/security/ddipe_detector.py`. Catches the "innocuous markdown contains an executable payload" pattern that's been observed in compromised skill imports in the wild.
- **Snyk skill scanner wrapper** under `scripts/security/mcp_scan_wrapper.py`. Opt-in (gated behind `SNYK_AGENT_SCAN_CONFIG`).
- **Agent-skill-evaluator wrapper** under `scripts/security/agent_skill_evaluator_wrapper.py`.

All three wrappers are soft-fail by default; `STRICT_SECURITY=1` escalates to hard-fail. Per the ADR's "block-by-default for net-new content, soft-fail for in-tree audit" contract.

## Privacy

The BOM does NOT contain any user data. It records public attributes of imported content (source URL, version, vetted date). No telemetry, no prompt bodies, no response bodies, no usage signals.

## How to inspect

```bash
cat docs/security/ai-bom.json | jq '.components[] | {kind, name, source, vetted_as_of}'
```

Or browse `docs/atlas/index.html` and look for the `vetted:` badge on each skill page.
