# 0011. Tier 2 -> Tier 1 adapter promotion criteria

## Status
Accepted (2026-05-24)

## Context

ADR-0005 defined three adapter tiers:
- **Tier 1 (full adapter)**: skills + rules + hooks + MCP + agents + commands. ~150-200 lines per adapter. v0.2: claude_code, codex, cursor, windsurf.
- **Tier 2 (skills + AGENTS.md)**: skills + rules + AGENTS.md, no hooks/MCP/agents/commands beyond what the agent natively supports. ~80-100 lines per adapter. v0.2: copilot, gemini_cli, aider, cline, pi.
- **Tier 3 (AGENTS.md only)**: generated by `agents_md.py`. Detection entry + project AGENTS.md materialization. No per-agent code.

The unanswered question from v0.1 carry-over: under what conditions does a Tier 2 adapter get promoted to Tier 1? Without criteria, promotion is ad-hoc and tier labels lose meaning over time.

## Decision

A Tier 2 adapter is promoted to Tier 1 when ALL three conditions hold:

1. **Adoption signal**: at least 3 named teammates report using the agent regularly (weekly+) as either their primary or one of their top-3 coding agents.
2. **Surface coverage**: the agent has a meaningfully distinct skill, agent, command, or MCP surface beyond what AGENTS.md provides. If everything the agent reads can be expressed in AGENTS.md, Tier 1 promotion adds no value.
3. **Maintenance commitment**: at least one named maintainer (individual handle or alias from OWNERS.md) commits to keeping the adapter green and reviewed within the playbook's 90-day decay window.

Promotion is a PR that:
- Moves the adapter from the Tier 2 entry to the Tier 1 entry in `scripts/install.py` AGENTS dict.
- Extends `scripts/adapters/<agent>.py` to materialize the additional surfaces (whichever of agents/commands/prompts/MCP the agent supports).
- Updates `scripts/test_adapters.py` to cover the new surfaces with idempotency + user-content-preservation checks.
- Updates `docs/tools/<agent>.md` to reflect the new tier and surface list.

## Demotion

A Tier 1 adapter is demoted to Tier 2 when ANY of the three Tier 1 conditions fails for two consecutive 90-day review cycles:
- No teammates report regular use (adoption decay).
- The agent dropped support for one of its distinct surfaces (e.g., deprecated skills).
- The maintainer commitment lapses (last_reviewed > 180 days, no responding owner).

Demotion is a PR that strips the adapter back to skills + AGENTS.md, retaining the detection entry and basic install path so existing users aren't broken.

## Antigravity case study (future application)

Antigravity is deferred to v0.3 per Q5 v0.2 lock. When a teammate first adopts it, the adapter ships as Tier 2 (skills + project AGENTS.md, the surfaces Google Antigravity exposes per docs.antigravity.google/docs/skills). It would qualify for Tier 1 promotion if/when:
- 3+ teammates use it regularly (condition 1).
- Google ships custom subagents or hooks (condition 2; currently no custom subagent surface).
- An owner commits to maintenance (condition 3).

## Reject if

- Promotion criteria become a procedural barrier that delays useful adoption work. Watch for: contributors wanting to add a surface but blocked on "do we have 3 adopters yet?" If observed, lower the adoption threshold or fold criteria 1 into the maintenance commitment.
- Demotion churn destabilizes user installs (e.g., a feature flag flipped in a coding agent triggers a Tier change). Defensive: require demotion PRs to ship migration notes so existing users see a clear upgrade path.

## Consequences

- Adapter tier becomes signal, not just decoration. A Tier 1 label means "this is a first-class supported integration with active maintenance"; Tier 2 means "skills + rules distribution available; deeper surfaces by request".
- New-agent introductions default to Tier 2 with a clear promotion path. Avoids the v0.1 trap of speculatively building Tier 1 adapters for hypothetical users.
- OWNERS.md (from Task 27) becomes load-bearing: criterion 3 requires a real owner entry.
