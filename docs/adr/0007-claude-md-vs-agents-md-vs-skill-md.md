# 0007. CLAUDE.md vs AGENTS.md vs SKILL.md (three separate formats)

## Status
Accepted (2026-05-24)

## Context

Three file formats overlap in scope but serve different purposes. Conflating them is a documented failure mode (ETH Zurich study, 2025).

| File | Purpose | Read by |
|---|---|---|
| `CLAUDE.md` | User-level memory + Claude Code project context | Claude Code only |
| `AGENTS.md` | Behavioral rules ("always/never do X"), always-on context | ~28 agents (mid-2026) |
| `SKILL.md` | Workflow orchestration ("how to do X"), invoked by name | Claude Code, Codex (others discover differently) |

## Decision

Strict separation:

- **CLAUDE.md**, Claude Code-specific user memory. Lives at `~/.claude/CLAUDE.md` (global) or per-project. Not generated by the playbook (the playbook generates user-level rules into AGENTS.md for cross-tool compatibility).
- **AGENTS.md**, concatenated behavioral rules. Generated by the playbook installer from `rules/*.md` files. Materialized at the project root (where ~28 agents read it) and in per-agent locations as needed.
- **SKILL.md**, per-skill workflow file with YAML frontmatter. Lives at `skills/<category>/<name>/SKILL.md` in the playbook; materialized as `~/.claude/skills/<name>/SKILL.md` and `~/.codex/skills/<name>/SKILL.md` by the respective adapters.

## Why

The ETH Zurich study cited in termdock.com 2025 showed:
1. Longer context files (300+ line CLAUDE.md with rules + workflows + style + architecture) hurt agent performance significantly.
2. Conflating workflow orchestration (skills) with behavioral rules (AGENTS.md) makes both worse.

Block/Goose's distinction between "Rules" (behavioral constraints) and "Recipes" (workflow orchestration) reinforces the same separation at 5,000-employee scale.

## Consequences

- The playbook authors think in three buckets: rules (constraints), skills (workflows), and hooks (automation that fires without agent involvement). Rules go in `rules/`, skills in `skills/<category>/<name>/`, hooks in `hooks/`.
- Contributors must understand which bucket their contribution belongs in. CONTRIBUTING.md and CONTEXT.md cover this.
- The installer translates: rules become `AGENTS.md` (concatenated), skills become `SKILL.md` files (one per skill, copied to agent-specific dirs), hooks become shell scripts plus registration in agent settings.

## What goes where

| Content | Lives in | Why |
|---|---|---|
| "Never use em dashes" | `rules/no-em-dashes.md` | Behavioral constraint, always-on |
| "Use VCS not GitHub" | `rules/VCS-not-github.md` | Behavioral constraint |
| "Steps to review a VCS PR with team rules" | `skills/engineering/code-review/SKILL.md` | Workflow, invoked by name |
| "Block direct pushes to develop" | `hooks/never-push-to-develop.sh` | Enforcement, no agent cooperation needed |
