# 0043. Marketplace distribution for selective skill install

## Status

Accepted (2026-05-28); shipped in v0.14.

## Context

The playbook is a clone-and-run kit. A user who wants a single skill (say, `code-review` or `handoff`) has to clone the whole repo and run `make install`. Other coding-agent ecosystems shipped lighter install paths between October 2025 and Q2 2026:

- Plugin marketplaces (Claude Code, Codex CLI, Cursor 2.5+, Gemini CLI extensions): one git repo exposes a `marketplace.json` catalog at a well-known path; users install bundles with `/plugin install <pkg>@<market>` or `gemini extensions install`.
- Vercel's `npx skills` CLI: per-skill install for 18 agents including Windsurf, Cline, Copilot, Pi, Kilo, Kiro, Goose, and others that don't have plugin marketplaces of their own.

The destination repo (the playbook's downstream public mirror, populated by `sync_distribution.py` per ADR-0042) is the natural place to expose these channels. The upstream source authors content once; the sync flows it to the destination; the destination publishes through both channels.

## Decision

Run three distribution channels concurrently:

| Channel | Audience | Granularity | Install command |
|---|---|---|---|
| A: Plugin marketplace | Claude Code, Codex, Cursor, Gemini CLI | Per-profile bundles (7 roles + 1 meta = 8 plugins) | `/plugin install <profile>@8v-coding-agents-playbook` |
| B: `npx skills` standalone | 18 agents including Windsurf | Per-skill | `npx skills add <destination-owner>/<destination-repo> --skill <slug>` |
| C: `make install` | Full-playbook install | Per-profile (existing) | `git clone && make install PROFILE=<name>` |

### Channel A: emit at sync time

A new `scripts/marketplace_emitter.py` runs after `sync_distribution.py` writes content. It reads `profiles/*.toml` + the destination's `base/`, then emits at the destination root:

```
.claude-plugin/marketplace.json         (Claude Code catalog)
.agents/plugins/marketplace.json        (Codex catalog)
.cursor-plugin/marketplace.json         (Cursor catalog)
gemini-extension.json                   (Gemini CLI extension; metadata-only)
plugins/<profile>/
  .claude-plugin/plugin.json            (with emittedBy marker)
  .codex-plugin/plugin.json
  .cursor-plugin/plugin.json
  skills/<category>/<slug>/             (materialized copies)
  rules/  hooks/  mcp/                  (materialized copies)
  hooks/hooks.json                      (aggregated from PLAYBOOK-HOOK-EVENT metadata)
  .mcp.json                             (aggregated from per-server configs)
  README.md                             (generated)
```

### Channel A: opt-in via manifest

Emit is opt-in. The `[marketplace]` block in the sync manifest carries `name`, `owner_name`, and optional `owner_email`/`homepage`. When absent, the sync ships content only. `--no-emit-marketplace` forces off even when configured. Default-off was chosen to keep first-time operators from being surprised by 40+ new files appearing at the destination.

### Channel A: per-profile granularity

7 role profiles + 1 synthetic meta-pack = 8 plugins. Per-skill granularity (~51 plugins) was rejected as too many for the agent's `/plugin` browse UI; per-content-type (7 plugins) was rejected because it loses skill-to-hook coherence.

### Channel A: materialize, don't symlink

Codex copies plugins to a cache at install time and does NOT follow directory symlinks. Symlinks would leave the cached plugin without its referenced SKILL.md files. Materializing (copying) content into each plugin directory makes the install path predictable across Claude Code, Codex, and Cursor. Trade-off is a bigger destination repo; git delta compression mitigates the wire cost.

### Channel A: per-agent schema variance

Each agent has a slightly different plugin manifest schema. Per `developers.openai.com/codex/plugins/build` and `code.claude.com/docs/en/plugin-marketplaces` (verified 2026-05-28):

- Claude Code / Cursor: `skills`/`agents`/`commands` accept string-or-array, `hooks` is a single hooks.json path, `mcpServers` is a single .mcp.json path.
- Codex marketplace: nested source `{"source": "local", "path": "..."}` with `policy.authentication: "ON_INSTALL"`. Plugin manifest accepts skills + apps + MCP, NOT sub-agents/commands/hooks (Codex strips these).
- Gemini CLI: single-extension-per-repo, no per-bundle catalog. The v0.14 manifest is metadata-only; Gemini users install skills via Channel B.

### Channel B: no new code

Vercel's `npx skills` walks any repo with SKILL.md files. The destination's `base/skills/<category>/<slug>/SKILL.md` layout works without changes. README documents the install commands.

### Channel C: stays canonical

`make install` continues to be the source of truth. Marketplace channels are projections of the same content tree.

## Consequences

### Pro

- One authoring surface (upstream source) feeds three install paths.
- Materialized content works across every installer's caching model.
- Derivation means a profile.toml edit propagates to the marketplace on next sync; no second authoring step.
- Each agent's native install command works (`/plugin install`, `gemini extensions install`, `npx skills add`).
- Windsurf is covered without Windsurf-specific code (Vercel's CLI does the work).
- The `emittedBy` marker in every emitted plugin.json lets orphan-cleanup safely distinguish ours from operator-authored plugins in the same `plugins/` directory.

### Con

- Content duplication at the destination: each profile gets its own materialized copy of every skill it references. Mitigated by git's delta compression.
- Four marketplace manifest formats to keep aligned. Schema churn is real (CC's 2.1.144 `./`-prefix change, Codex's nested-source shape both required earlier emit fixes). Centralised in the emitter; one fix migrates all manifests.
- Codex's manifest omits sub-agents / commands / hooks even when the profile lists them; per-agent rendering handles this in the emitter.

### Considered alternatives

1. Per-skill plugins (~51). Rejected: too many to browse, maintenance overhead per release.
2. Per-content-type plugins (7: skills, rules, hooks, mcp, agents, commands, prompts). Rejected: loses skill-to-hook coherence.
3. Symlinks instead of copies. Rejected after the Codex cache install verification.
4. Hand-authored marketplace metadata. Rejected: schema churn would multiply the maintenance surface.
5. Default-on emit. Rejected for v0.14: first-time operators get unexpected files. Future v0.15 may flip after the workflow stabilises.

## Failure modes

- Reserved marketplace name collision: emit exits 5. Reserved list includes `claude-plugins-official`, `anthropic-*`, `claude-code-*`, `agent-skills`, `knowledge-work-plugins`, `life-sciences`, `claude-for-legal`, `claude-for-financial-services`, `financial-services-plugins`.
- Symlink target escapes base/ or destination: exit 5 (path-traversal block at `_within(link, destination)` AND `_within(target, base_dir)`).
- Profile slug contains `..`, `/`, backslash, or non-`[A-Za-z0-9_./-]` characters: exit 5.
- Two profiles declare the same `name`: exit 5.
- A role profile collides with the synthetic `meta` name: exit 5.
- Destination has no `base/` directory: exit 1 (content sync probably failed).
- Profile references a non-existent skill / rule / hook / MCP / agent / command / prompt: warning + prune (not fatal). Scrub patterns in v0.13 rewrite content in profile TOMLs but don't rename filenames in base/; fail-fast would block every cron fire on this expected drift.

## Acceptance criteria

- [x] `scripts/marketplace_emitter.py` exists and is idempotent.
- [x] Emitter generates Codex `.agents/plugins/marketplace.json` + per-plugin `.codex-plugin/plugin.json` with the nested `source` shape + `ON_INSTALL` authentication.
- [x] Emitter generates Cursor `.cursor-plugin/marketplace.json` and per-plugin `.cursor-plugin/plugin.json` (currently identical to Claude shape per the v0.14 schema; future divergence point preserved).
- [x] Emitter generates one `gemini-extension.json` at repo root (metadata-only for v0.14).
- [x] Plugin content materialized (copied) into each plugin directory.
- [x] hooks.json + .mcp.json aggregated per plugin where the profile lists hooks/MCP.
- [x] Path-traversal guards reject targets outside `base/` AND destination root.
- [x] `sync_distribution.py` accepts `--no-emit-marketplace` to opt out.
- [x] Reserved marketplace names cause emit to exit 5.
- [x] `emittedBy` marker required for orphan-cleanup deletion.
- [x] Tests cover schema, symlink/materialize safety, idempotency, per-agent variance, missing-reference warn-prune, slug validation, collision detection (55 tests).
- [x] README + runbook document the three install commands (G1 + G2).
- [ ] `claude plugin validate --strict` integration as part of `make check` (deferred to v0.15).

## Migration

This is a v0.14 addition; nothing migrates. First emit produces the new layout at the destination on next sync after the operator adds `[marketplace]` to their manifest.
