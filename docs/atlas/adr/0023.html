# 0023. Lifecycle ownership model + anchored-fs default scope

## Status

Accepted (2026-05-25); landed in v0.3 after adversarial Codex review

## Context

The adversarial Codex review of the v0.3 release branch surfaced three high-severity design concerns that survived the earlier P1/P2 fix pass:

1. **`make remove` could delete user-authored content.** The lockfile recorded every adapter-written file (including managed-block files like `~/.codex/AGENTS.md` that mix playbook content with user-authored text). `cmd_remove` walked the lockfile and unlinked everything, losing user content.
2. **Lockfile coverage gap.** The hard-coded glob table missed Cursor project-level paths, MCP configs, skill `references/`/`scripts/`, target-scoped paths for Windsurf, etc. `make status` reported clean while drift existed.
3. **anchored-fs default-allowed-dir was `~`.** Combined with the bundle's passthrough `write_file` / `create_directory` / `move_file` tools, this gave any MCP client write access to the entire home directory.

## Decision

### Ownership model for lifecycle paths

`ADAPTER_DEST_PATHS` entries gain an `ownership` field. Each entry is `(dir, glob, ownership)`:

- `"owned"`: the playbook fully owns the file (skill dirs, agent definitions, hook scripts, etc.). Safe to unlink on `make remove`, IF the current hash matches what the lockfile recorded.
- `"managed"`: the file mixes playbook + user content (`~/.codex/AGENTS.md`, `~/.claude.json`, `~/.cursor/mcp.json`, etc.). `make remove` NEVER unlinks these. Users must clean managed-block markers via the adapter's own remove flow.

Lockfile schema: each path now records `{"sha256": ..., "ownership": ...}` instead of a bare hash. Backward-compatible: a bare string is read as `{"sha256": <str>, "ownership": "owned"}`.

### Expanded path coverage

`ADAPTER_DEST_PATHS` now includes:

- Skill support dirs (`*/references/**/*.md`, `*/scripts/**`)
- Managed-block files (`~/.claude/settings.json`, `~/.claude.json`, `~/.codex/AGENTS.md`, `~/.codex/config.toml`, etc.)
- Cursor user-level + project-level (`~/.cursor/rules/`, `{TARGET}/.cursor/rules/`, `~/.cursor/mcp.json`)
- Windsurf target-scoped paths (`{TARGET}/.windsurf/skills/`, `{TARGET}/.windsurfrules`)
- Pi prompts (`~/.pi/agent/prompts`)

The `{TARGET}` placeholder is resolved at scan time from the `target` argument or the lockfile's recorded target.

The plan's full per-adapter-emitted manifest (each adapter returns the exact list of paths it wrote) remains a v0.4 work-stream. This commit closes the worst gaps; deeper coverage is incremental.

### anchored-fs default scope

`mcp/anchored-fs/server.json` default `--allowed-dir` changes from `~` to a new placeholder `{{PLAYBOOK_TARGET}}` that the loader resolves at install time:

- With a target set: `--allowed-dir <target-project>` (narrow, default)
- Without a target: falls back to `$HOME` (legacy behaviour; opt-in by simply not setting a target)

The loader's `expand_agent_shared_placeholder()` now takes an optional `target` argument; all adapter call sites pass `_loader.get_target()`. The server.json file includes a `_comment_scope` field documenting the trade-off so users see the security rationale when editing.

## Consequences

- `make remove` is now safe on managed-block files. Users see clear "Skipped managed-block file" output rather than silent data loss.
- `make remove` refuses to unlink user-edited owned files (hash mismatch). Users see "SKIP edited (hash mismatch)" output.
- Lockfiles generated by v0.3+ are richer; v0.2.x or pre-fix lockfiles still parse (backward-compatible reader handles bare strings).
- Anchored-fs sessions narrow filesystem write scope to the target project by default. Users who need home-wide access (e.g. for cross-repo workflows) add another `--allowed-dir` entry by editing the materialized config; the upstream README documents this.

## Future work (deferred to v0.4+)

- Per-adapter `installed_paths()` function so the lockfile is generated from adapter-emitted manifests rather than scanned via `ADAPTER_DEST_PATHS`.
- Anchored-fs: split passthrough write/move tools into a separate opt-in flag rather than enabling them with the rest of the bundle.
- `make remove --force` flag to bypass the hash check (current behaviour requires manual unlink).
