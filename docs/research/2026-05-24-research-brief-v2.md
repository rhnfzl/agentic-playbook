## Personal-to-Team Transition Lessons

### Patterns that held up (with primary sources)

1. **Rules-as-code, committed to the repo.** The pattern most consistently endorsed: commit `.cursor/rules/`, `AGENTS.md`, or `CLAUDE.md` to the shared repo, and every developer gets the same rules automatically. "A prompt on an individual machine is a personal productivity hack. The same prompt in the team's repository is infrastructure." (Martin Fowler, martinfowler.com/articles/reduce-friction-ai) This held up even at Stripe's unattended agent scale.

2. **Directory-scoped rules, not monolithic global rules.** Stripe confirmed that loading all rules globally fills the context window and degrades performance. The pattern that works: put rules closest to the code they govern, let agents pick them up as they traverse the filesystem. (blog.bytebytego.com/p/how-stripes-minions-ship-1300-prs). IBM's governance writeup also identifies this as the correct hierarchy: global rules for security/cross-cutting, workspace rules for project-specific, project rules override global. (ibm.com/think/insights/standardize-ai-code-generation)

3. **Failure-as-feedback loop.** Sourcegraph CEO Quinn Slack: "If [the agent] makes a mistake, then they'll go and update [the rules] and try to get it to be a flywheel." (stackoverflow.blog/2026/03/26/coding-guidelines-for-ai-agents) Airbnb validated this at scale, brute-force retry with structured error feedback outperformed upfront prompt engineering. Rules files should be living documents updated on every agent mistake, not written once.

4. **Separate "what" (rules) from "how" (skills/recipes).** Rules files tell agents what standards to follow. Skills/Recipes tell agents what sequence of steps to execute. Conflating the two into one large file consistently fails. Goose distinguishes "Rules" (behavioral constraints) from "Recipes" (workflow orchestration). Claude Code distinguishes `CLAUDE.md` (project context) from `SKILL.md` (task workflow). This separation held up across tools. (nickyt.co/blog/what-makes-goose-different; termdock.com/blog/skill-md-vs-claude-md-vs-agents-md)

5. **Single source of truth with symlinks for multi-tool.** The AGENTS.md → symlink pattern ("ln -sfn AGENTS.md .github/copilot-instructions.md") is widely adopted as the lowest-friction way to maintain one canonical file while satisfying per-tool format requirements. (medium.com data-science-collective CLAUDE.md guide) Ruler CLI automates this pattern. Both Kiro and Windsurf confirm AGENTS.md interop.

### Patterns that broke (with primary sources)

1. **Monolithic CLAUDE.md or AGENTS.md files.** ETH Zurich study (cited in termdock.com blog) showed directly: longer context files hurt performance. Agents follow some instructions, ignore others. "300+ line CLAUDE.md files with coding standards, architecture docs, workflow instructions, and style guides" is the most common failure mode. Practical ceiling: one screen. (termdock.com/blog/skill-md-vs-claude-md-vs-agents-md)

2. **LLM-generated context files.** ETH Zurich specifically tested this. LLM-generated context files are verbose, generic, and full of restated information the agent can already see in the codebase. They produce worse results than hand-written files. Write context files yourself. (termdock.com, same source)

3. **Rules without model-tier awareness.** Cursor community forum (December 2025 thread): "rules is followed well by certain models, especially Claude's and latest GPT ones. We had many developers using the Auto mode... these guys always had trouble following rules." Cheaper/smaller models in a team setting skip rules. An 8-engineer team where 3 use Auto mode gets inconsistent rule adherence. Rules that "work" individually may fail in team deployments because of model tier variance. (forum.cursor.com/t/anyone-here-using-cursor-with-a-team)

4. **One-size-fits-all org rules.** Packmind's research identifies this as "Error #3": "A single context file for an entire organization... either becomes uselessly generic or dangerously incorrect for at least two of the three" teams. Jellyfish 2025 data: 48% of companies run two or more AI coding tools in parallel. (packmind.com/context-engineering-ai-coding/how-to-implement-context-engineering). A monorepo with two very different subprojects (like team's AI Backend + MCP server) needs directory-scoped rules, not a single AGENTS.md.

5. **Rules without validation/decay prevention.** Packmind identified four symptoms of context drift compounding over time: pattern violation (agents suggest deprecated APIs), architectural drift (locally coherent but globally inconsistent), staleness (instructions no longer reflect the codebase), inconsistency between agents/repos. Most teams write a rules file once and forget it. Without a review cadence or CI check, rules decay within weeks. (packmind.com/context-engineering-ai-coding/best-context-engineering-tools)

6. **Cursor Team Rules via dashboard, no automation path.** Cursor forum thread on team usage: Teams trying to enforce rules across the org discover that Team Rules have no API/CLI. Each rule must be managed in the dashboard. Changes don't propagate via git. This creates a split between version-controlled project rules (in `.cursor/rules/*.mdc`) and centrally-managed Team Rules (dashboard-only), and the two systems can diverge silently.

## Failure Modes from Real Deployments

Primary sources only. Marketing inference excluded.

- **Context window exhaustion from over-broad global rules.** Stripe explicitly stated they use global rules "very judiciously" because loading all rules globally fills the context window before the agent starts work. (ByteByteGo, Stripe Minions writeup, 2026). The failure mode: a team writes comprehensive rules, they get injected on every request, performance degrades, developers start ignoring rule failures as "normal."

- **Model-tier compliance gap.** Cursor forum (December 2025, 8-engineer team thread): Rules that work with Claude Sonnet/Opus or GPT-5 reasoning models are silently ignored by cheaper "Auto" mode or smaller models. A team where developers use different model tiers gets inconsistent rule adherence with no visibility into why. Solution attempted by affected team: enforce a minimum model tier org-wide. Cursor Enterprise now supports model controls for this.

- **Rules not following tool upgrades.** Cursor forum (December 2025): "I feel like rules have been worse and worse. The option doesn't even appear in '@' anymore." Rule syntax that works in one Cursor version breaks silently in the next. Rules files written against `.cursorrules` (deprecated) vs. `.cursor/rules/*.mdc` (current) show this lifecycle problem. Teams that don't own rule maintenance fall behind tool updates.

- **Team coordination problems amplified, not solved.** LinkedIn post (ShriKant Vashishtha, citing a real team failure): "A team got AI coding tools for every developer. Leadership was excited. Sprint 1 ended. Not a single story was done." The team had coordination problems (siloed developers, testing as a separate downstream activity) that AI tools amplified. Rules alone don't fix team process debt.

- **Context file format fragmentation.** Even with AGENTS.md convergence, tools still vary: Cursor uses YAML frontmatter in `.mdc` files; Windsurf has a 6,000-char global limit; GitHub Copilot reads `AGENTS.md` but also `.github/copilot-instructions.md` and org-level instructions; Kiro has inclusion modes (always/conditional/manual) that AGENTS.md doesn't support. Teams that symlink naively lose tool-specific features (Cursor's `description`-based routing, Kiro's `fileMatch` patterns).

- **Packmind's four documented drift symptoms** (January 2026, Mike Mason analysis): pattern violation (deprecated API suggestions), architectural drift (locally coherent, globally inconsistent decisions), staleness (instructions no longer reflect actual codebase), inter-agent inconsistency (different tools producing conflicting code). All four compound over time without active governance.

- **LLM-generated rule files backfire.** ETH Zurich study (cited in termdock.com): LLM-generated context files produce worse agent performance than hand-written ones. Generic, verbose, restating information the agent can already see. Teams that automate rule creation with "generate our CLAUDE.md from codebase" get worse results than teams that write rules manually based on observed failures.

- **One global rules file for heterogeneous teams.** Packmind, IBM: a single AGENTS.md for a frontend React team and a backend Go microservices team is either too generic (ignored) or too specific for one team (actively wrong for the other). 48% of companies run 2+ AI tools in parallel; one-size-fits-all rules compound with one-size-fits-all tool selection.

## Revised Recommendations

The v2 research materially changes two v1 recommendations and adds three new ones.

**What changes from v1:**

First, the "Windsurf-first team" framing (v1 gap #2) needs updating. Windsurf was acquired by Cognition in July 2025; its enterprise trajectory is now tied to Cognition's roadmap, and regulated-industry teams are scrutinizing whether previously-negotiated DPAs carry through the acquisition. For the team team today (Windsurf + Claude Code + Codex), the practical architecture remains correct, but Windsurf's long-term enterprise tier should be monitored and a fallback evaluated quarterly.

Second, the agent landscape is wider than v1's four tools, but AGENTS.md convergence is the key structural fact: as of mid-2026, Claude Code, Codex, Cursor, Windsurf, Copilot, Gemini CLI, Kiro, Junie, Goose, Aider, Cline, and Roo Code all read AGENTS.md at the repo root. A canonical `AGENTS.md` per subproject (AI Backend + MCP) is genuinely cross-tool now, not just aspirationally. The Ruler CLI handles distribution to all 20+ tools; Packmind adds governance if team grows to a stage where drift detection and audit trails are required.

Third, the Stripe finding on rule scoping is the most important new finding for architecture: do NOT write a monolithic AGENTS.md that applies to both AI Backend and MCP simultaneously. The two subprojects have different rules, different MCP boundaries, and different vocabulary. Each subproject should own its AGENTS.md in its repo root, with workspace-level cross-cutting rules distributed via Ruler from the `team-agents` repo into each subproject's git history. This matches Stripe's "global rules used very judiciously; most rules scoped to subdirectory" finding.

Fourth, the Cursor Team Rules API gap is confirmed with no workaround. For team's 10-engineer scale, the practical mitigation is: (a) manage team rules as version-controlled `.cursor/rules/*.mdc` files committed to each subproject repo, and (b) reserve Cursor Enterprise Team Rules dashboard only for rules that are truly universal and rarely change (e.g., "never push to develop"). Do not try to manage per-feature or per-file rules through the dashboard, they won't survive git workflows.

Fifth, rules decay is the #1 long-term risk. Block's Goose approach (Recipes with explicit session export and metadata) and Packmind's drift detection are the two production-validated solutions. For team's current scale, the pragmatic minimum is: every SKILL.md has `last_reviewed: YYYY-MM-DD` in frontmatter, and a CI job warns (not fails) when any skill exceeds 90 days without review. Promote to blocking failure at 180 days.
