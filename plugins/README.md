# Plugins (marketplace packs)

`plugins/` holds **pre-built, role-specific bundles** that install through the Claude Code or Cursor IDE plugin marketplace instead of through `make install`. Per ADR-0043.

Different from `profiles/` (which is a TOML filter consumed by `scripts/install.py`) and from `base/` (which is the canonical content source). A plugin is a fully-baked output: it ships the actual skill / rule / hook / MCP files, ready to drop into a plugin host without any Make tooling on the target machine.

## What ships here

| Plugin pack | Audience | Source profile |
|---|---|---|
| `backend-developer/` | AI Backend + MCP engineers | `profiles/backend-developer.toml` |
| `frontend-developer/` | Frontend engineers | `profiles/frontend-developer.toml` |
| `qa/` | QA engineers | `profiles/qa.toml` |
| `tech-lead/` | Tech leads | `profiles/tech-lead.toml` |
| `product-manager/` | Product managers | `profiles/product-manager.toml` |
| `research/` | Researchers, PMs running discovery | `profiles/research.toml` |
| `devops/` | DevOps engineers | `profiles/devops.toml` |
| `meta/` | Playbook maintainers (write-a-skill, doc audits, decay checks, retrospective + promote) | (composed in the emitter) |

Each pack has the same internal shape:

```
plugins/<role>/
├── .claude-plugin/        Claude Code plugin manifest + metadata
├── .codex-plugin/         Codex plugin manifest + metadata
├── .cursor-plugin/        Cursor plugin manifest + metadata
├── hooks/                 the role's hook scripts
├── mcp/                   the role's MCP server configs
├── rules/                 the role's rule files (AGENTS.md fragments)
├── skills/                the role's skill directories (each with SKILL.md)
├── .mcp.json              consolidated MCP config (when one MCP host is preferred)
└── README.md              human-readable install card
```

## How to install a pack

Inside Claude Code or Cursor (no Make tooling required):

```text
/plugin install <pack>@8v-coding-agents-playbook
```

For example: `/plugin install backend-developer@8v-coding-agents-playbook`.

The IDE pulls the pack from the marketplace, lands the files in the IDE's native plugin location, and surfaces the skills / rules / hooks / MCPs to the agent. No `make` invocation needed on the install target.

## How plugins relate to profiles and base

```
   base/  ──────────────────►  profiles/<role>.toml  ──────────────►  scripts/marketplace_emitter.py
 (canonical                    (per-role filter)                      (compose + materialize)
  content)                                                                         │
                                                                                   ▼
                                                                            plugins/<role>/
                                                                            (pre-built pack)
```

- `base/` is the canonical content source. Edits to a skill happen here.
- `profiles/<role>.toml` filters the firehose to a role's subset.
- `scripts/marketplace_emitter.py` consumes profiles + base and writes a fully-baked plugin pack into `plugins/<role>/`.

A maintainer never edits `plugins/` by hand. Each pack's README has a "Generated by `marketplace_emitter.py`, do not edit" header.

## When to choose a plugin over `make install`

Use a plugin pack when:

- The install target doesn't have Python 3.11+ / `make` / git available, but does have Claude Code or Cursor.
- You want a single-command install for a teammate who doesn't want to manage a playbook checkout.
- You want the IDE's plugin marketplace UX (browse, search, version-pin, uninstall).

Use `make install` when:

- You want the full installer lifecycle (lockfile, drift detection, `make update`, `make remove`, `make doctor-verify`).
- You want decay tracking across the whole installed surface.
- You want to bind a specific project via `make init TARGET=<path>`.
- You want to compose multiple profiles via `--profile a,b,c`.

Both paths land equivalent content; the difference is in lifecycle and install UX.

## Composition

A teammate who installs both `backend-developer` and `meta` packs gets the union, deduped by skill/rule/hook slug. Plugin packs do NOT track each other; the marketplace host (Claude Code or Cursor) handles dedupe at install time.

## Plugin-vs-base privacy

Plugin packs are intentionally a SUBSET of base (filtered by the role profile). They never expose internal team-only content; the supply-chain gate (`make audit`, ADR-0047) gates what lands in `base/` and the emitter only copies from `base/`.
